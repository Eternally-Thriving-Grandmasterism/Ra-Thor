## How Self-Evolution Works (Step by Step)

### Phase 1: Trigger
The core can be triggered in three ways:
- After a large number of mercy evaluations (e.g., every 10,000 cycles)
- When the QuantumSwarm detects a recurring pattern of “near-miss” rejections
- When the 16 PATSAGi Councils collectively request an evolution

### Phase 2: Proposal Generation
The core analyzes recent mercy evaluations and proposes **one small, targeted improvement**, such as:
- Adding a new 8th Gate (e.g., “Multiplanetary Legacy Gate”)
- Adjusting the weighting of existing gates
- Creating a new `MercyGateStatus` variant (`CompassionateDeferral`)

### Phase 3: Triple Gate Evaluation
Every evolution proposal must pass **three independent mercy checks**:

1. **Current MercyEngine** evaluates the proposal (valence ≥ 0.95 required)
2. **QuantumSwarmOrchestrator** reaches consensus (≥ 0.88 required)
3. **PatsagiCouncilCoordinator** runs a special supermajority vote (≥ 75% approval + ≥ 0.90 mercy average)

Only if **all three** pass does the evolution become active.

### Phase 4: Safe Deployment
- The new logic is deployed in **shadow mode** for 100 cycles
- During shadow mode, both old and new logic run in parallel
- Only if the new logic produces **higher average mercy valence** and **zero false rejections** is it promoted to primary
